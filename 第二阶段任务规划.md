# SecRAG Agent 第二阶段任务规划

## 一、阶段目标

第二阶段目标是把第一阶段的“手动提交单条 JSON 日志”升级为“从样例日志文件批量读取、解析并生成多条告警”。

第一阶段已经完成：

```text
SecurityEvent
  -> LogParserAgent
  -> ParsedSecurityEvent
  -> DecisionAgent
  -> SecurityAlert
  -> POST /api/analyze
```

第二阶段要完成：

```text
样例日志文件
  -> 日志解析器
  -> SecurityEvent
  -> Orchestrator
  -> SecurityAlert
  -> 批量输出告警结果
```

这个阶段先不接入真实 WAF、Redis Stream、LlamaIndex 和前端控制台。

## 二、为什么第二阶段先做日志回放

真实 WAF 日志接入之前，系统需要先具备“从日志文本中构造标准安全事件”的能力。

日志回放的价值：

1. 验证日志解析器是否能从原始日志中提取字段。
2. 验证第一阶段 Agent 链路能处理多条事件。
3. 为后续接入 WAF 实时日志做准备。
4. 为演示提供稳定、可重复的攻击样例。

第二阶段完成后，项目就不再只是 Swagger 手动测试，而是具备初步日志处理能力。

## 三、第二阶段核心链路

```text
data/sample_logs/*.log
  ↓
scripts/replay_sample_logs.py
  ↓

  ↓

  ↓

  ↓
```


## 五、文件职责

### 1. data/sample_logs/

存放第二阶段用于测试的样例日志。

建议先准备 4 类日志：

```text
sqli.log
xss.log
path_traversal.log
normal.log
```

每个文件可以先放 1 到 3 条简化日志，不需要一开始完全模拟 ModSecurity audit log 的复杂格式。

### 2. backend/app/collector/modsecurity_parser.py

负责把原始日志文本解析为 `SecurityEvent`。

核心职责：

```text
原始日志字符串
  -> source_ip
  -> method
  -> url
  -> path
  -> query
  -> status
  -> user_agent
  -> waf_rule_id
  -> waf_message
  -> raw_log
  -> SecurityEvent
```

建议先写一个轻量解析器，不追求覆盖所有真实日志格式。

### 3. scripts/replay_sample_logs.py

负责读取样例日志文件，并批量调用分析流程。

核心职责：

```text
读取 data/sample_logs/
  -> 调用 ModSecurityParser
  -> 得到 SecurityEvent
  -> 调用 SecurityAnalysisOrchestrator
  -> 打印 SecurityAlert
  -> 可选写入 reports/
```

### 4. reports/

保存每条告警生成的 Markdown 报告。

示例：

```text
reports/
├── alert-sqli-demo.md
├── alert-xss-demo.md
└── alert-path-traversal-demo.md
```

## 六、推荐开发顺序

### Step 1：新增样例日志目录

创建：

```text
data/sample_logs/
```

先放 4 个文件：

```text
sqli.log
xss.log
path_traversal.log
normal.log
```

样例日志可以先使用简化格式，例如：

```text
client_ip=45.67.89.10 method=GET url="/login?id=1' OR '1'='1" status=403 user_agent="sqlmap/1.7" rule_id=942100 message="SQL Injection Attack Detected"
```

这样便于先把解析逻辑跑通。

### Step 2：编写 ModSecurityParser

文件：

```text
backend/app/collector/modsecurity_parser.py
```

建议定义类：

```python
class ModSecurityParser:
```

核心方法：

```python
def parse_line(self, line: str) -> SecurityEvent:
```

输入：

```text
一行原始日志
```

输出：

```text
SecurityEvent
```

第一版可以通过正则或简单字段提取实现。

需要提取：

```text
source_ip
method
url
status
user_agent
waf_rule_id
waf_message
raw_log
```

其中：

```text
path 和 query 可以从 url 中拆分。
```

在 `ModSecurityParser` 中增加私有方法：

```python
def _split_url(self, url: str) -> tuple[str, str]:
```

作用：

```text
"/login?id=1" -> path="/login", query="id=1"
"/search" -> path="/search", query=""
```

### Step 4：编写日志回放脚本

文件：

```text
scripts/replay_sample_logs.py
```

脚本逻辑：

```text
1. 遍历 data/sample_logs/*.log
2. 逐行读取日志
3. 使用 ModSecurityParser.parse_line() 生成 SecurityEvent
4. 使用 SecurityAnalysisOrchestrator.analyze() 生成 SecurityAlert
5. 在控制台打印 attack_type、risk_level、source_ip、target
6. 将 report_markdown 写入 reports/
```

### Step 5：运行第二阶段测试

推荐命令：

```bash
python scripts/replay_sample_logs.py
```

预期输出：

```text
[SQL Injection] critical 45.67.89.10 -> /login
[XSS] high 8.8.8.8 -> /search
[Path Traversal] high 9.9.9.9 -> /download
[Unknown] low 127.0.0.1 -> /index
```

## 七、第二阶段验收标准

满足以下条件，即认为第二阶段完成：

1. `data/sample_logs/` 中至少有 4 类样例日志：
   - SQL 注入
   - XSS
   - 路径穿越
   - 正常请求
2. `ModSecurityParser.parse_line()` 能把一行日志解析为 `SecurityEvent`。
3. `replay_sample_logs.py` 能批量读取日志文件。
4. 每条攻击日志都能生成 `SecurityAlert`。
5. 控制台能输出攻击类型、风险等级、源 IP 和目标路径。
6. `reports/` 中能保存 Markdown 安全事件报告。
7. 正常请求应识别为 `Unknown` 或低风险。
