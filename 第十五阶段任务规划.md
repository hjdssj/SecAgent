# 第十五阶段任务规划：外部 API 接入优先级与实施路线

## 一、阶段定位

第十五阶段的目标是规划 SecRAG Agent 后续真正需要接入的外部 API。

当前项目已经具备：

```text
WAF 日志接入
Redis Stream 实时事件流
攻击识别
RAG 检索
企业上下文
AutoTriage 自动分流
告警持久化
前端 SOC 控制台
环境变量集中配置
```

但当前仍有一些能力没有真正接入外部服务：

```text
LLM 报告生成
真实 IP 威胁情报
真实 URL / 域名信誉查询
向量检索 Embedding
最新漏洞情报搜索
通知联动
```

本阶段只做规划，不做代码实现。

核心原则：

```text
先接最能提升产品价值的 API
先接低复杂度、低风险、可测试的 API
避免一次性接太多外部服务
所有 API 都必须可配置、可关闭、可降级
```

## 二、API 接入优先级

建议优先级：

```text
P0: LLM API
P1: AbuseIPDB 威胁情报 API
P2: VirusTotal 威胁情报 API
P3: Embedding API / 本地 Embedding + Milvus
P4: 搜索 API
P5: 通知 API
```

暂不建议接：

```text
股票行情 API
招投标 API
DocMind 文档解析 API
JWT 登录认证
```

原因：

```text
这些 API 和当前 SecAgent 主线关系弱
会让项目发散
当前阶段更应该围绕 SOC 告警分析闭环增强
```

## 三、P0：LLM API

### 作用

LLM API 是下一步最关键的外部 API。

用途：

```text
综合多源证据
生成 Markdown 安全分析报告
解释攻击判断
结合 RAG 引用生成处置建议
结合企业上下文分析业务影响
辅助人工复核 needs_review 告警
```

### 当前可用配置

`.env` 中已有：

```text
DASHSCOPE_API_KEY
DASHSCOPE_BASE_URL
OPENAI_MODEL
```

当前也规划了：

```text
LLM_ENABLED
LLM_PROVIDER
LLM_BASE_URL
LLM_MODEL
LLM_TIMEOUT_SECONDS
LLM_MAX_TOKENS
LLM_TEMPERATURE
LLM_ONLY_FOR_REVIEW
LLM_MIN_RISK_LEVEL
```

建议后续补：

```text
LLM_API_KEY
```

或者在代码中支持：

```text
LLM_API_KEY 优先
DASHSCOPE_API_KEY 作为 fallback
```

### 调用时机

不建议每条告警都调用 LLM。

建议策略：

```text
fast - 不调用 LLM
enriched - 默认不调用 LLM
deep - 调用 LLM
requires_human_review=true - 调用 LLM
```

初版策略：

```text
LLM_ENABLED=true
且
(
  analysis_mode == deep
  或 requires_human_review == true
  或 risk_level >= LLM_MIN_RISK_LEVEL
)
```

### 输入

LLM 输入不直接给无限原始日志。

输入应为结构化摘要：

```text
alert_id
attack_type
risk_score
risk_level
source_ip
target
confidence
evidence
mitre_attack
RAG 引用
威胁情报摘要
历史行为摘要
企业上下文
AutoTriage 结果
原始日志摘要
```

### 输出

输出 Markdown：

```text
结论
关键证据
攻击链分析
业务影响
处置建议
引用来源
不确定性说明
```

### 降级策略

LLM 调用失败时：

```text
保留原模板报告
记录 llm_error
不中断告警生成
不中断 Redis / DB 链路
```

### 验收标准

```text
LLM 默认关闭
开启后 deep / needs_review 告警可生成 LLM 报告
报告包含证据、引用和处置建议
失败时回退模板报告
测试不调用真实 LLM API
```

## 四、P1：AbuseIPDB 威胁情报 API

### 作用

AbuseIPDB 适合做真实 IP 信誉查询。

用途：

```text
查询源 IP 是否被举报
获取 abuse confidence score
获取国家、ISP、使用类型
判断是否疑似扫描器、代理、云主机、恶意来源
补充 evidence
影响 risk_score
辅助 AutoTriage
```

### 当前可用配置

`.env` 中已有：

```text
ABUSEIPDB_API_KEY
```

建议新增：

```text
THREAT_INTEL_ABUSEIPDB_ENABLED=false
ABUSEIPDB_BASE_URL=https://api.abuseipdb.com/api/v2
ABUSEIPDB_TIMEOUT_SECONDS=10
ABUSEIPDB_MAX_AGE_DAYS=90
```

### 调用时机

不建议低风险请求都查。

建议：

```text
analysis_mode == deep
或 risk_level >= high
或 requires_human_review == true
```

也可以增加缓存：

```text
intel:abuseipdb:{source_ip}
```

缓存时间：

```text
6 到 24 小时
```

### 输入

```text
source_ip
```

### 输出映射

映射到当前 `ThreatIntelResult`：

```text
source_ip
reputation
risk_score
tags
source=abuseipdb
description
```

### 降级策略

```text
API 失败时回退 local_mock
超时时不阻塞主链路
记录 evidence 或 metadata 说明情报不可用
```

### 验收标准

```text
开启 AbuseIPDB 后高风险 IP 可以查到真实情报
API 失败不会影响告警生成
相同 IP 重复查询可走缓存
测试使用 fake HTTP client
```

## 五、P2：VirusTotal API

### 作用

VirusTotal 适合补充：

```text
IP 信誉
域名信誉
URL 信誉
安全厂商检测结果
恶意标签
```

### 当前可用配置

`.env` 中已有：

```text
VIRUSTOTAL_API_KEY
```

建议新增：

```text
THREAT_INTEL_VIRUSTOTAL_ENABLED=false
VIRUSTOTAL_BASE_URL=https://www.virustotal.com/api/v3
VIRUSTOTAL_TIMEOUT_SECONDS=10
```

### 调用时机

建议晚于 AbuseIPDB。

初版只查：

```text
source_ip
```

后续再扩展：

```text
domain
url
file hash
```

### 输出映射

同样映射到：

```text
ThreatIntelResult
```

如果 AbuseIPDB 和 VirusTotal 都启用：

```text
ThreatIntelAggregator
  -> 合并多个来源
  -> 取最高风险分
  -> 保留来源列表
```

### 降级策略

```text
API 失败时不阻塞主链路
保留 AbuseIPDB 或 local_mock 结果
```

### 验收标准

```text
可以查询 IP 信誉
可以合并到告警 evidence
失败时可降级
测试使用 fake HTTP client
```

## 六、P3：Embedding API / Milvus 向量检索

### 作用

当前 RAG 已经有：

```text
Query Rewrite
BM25Retriever
HybridRetriever
VectorRetriever placeholder
```

下一步如果要提升语义召回，可以接：

```text
Embedding 模型
Milvus
```

用途：

```text
安全术语语义召回
攻击描述相似匹配
CVE / ATT&CK / 修复建议语义检索
BM25 + Vector 混合召回
```

### 当前可用配置

`.env` 中已有：

```text
MILVUS_HOST
MILVUS_PORT
```

建议新增：

```text
EMBEDDING_ENABLED=false
EMBEDDING_PROVIDER=dashscope
EMBEDDING_API_KEY=
EMBEDDING_MODEL=
EMBEDDING_DIM=
MILVUS_COLLECTION=secagent_knowledge
```

### 调用时机

不是实时每条日志都重新生成 embedding。

正确方式：

```text
知识库离线入库时生成 embedding
告警查询时只对 query 生成 embedding
HybridRetriever 合并 BM25 和 Vector 结果
```

### 验收标准

```text
知识库可以写入 Milvus
查询可以返回 vector retrieval result
HybridRetriever 可以合并 BM25 和 Vector
RAG 引用保留 score 和 source
```

## 七、P4：搜索 API

### 作用

搜索 API 用于补充最新漏洞和公开情报。

可用配置：

```text
BOCHA_API_KEY
SERPER_API_KEY
```

用途：

```text
搜索最新 CVE
搜索厂商安全公告
搜索近期攻击活动
辅助 0day / 在野利用判断
```

### 为什么不优先接

```text
搜索结果不稳定
会增加延迟
会增加幻觉风险
需要更多结果过滤和来源校验
```

建议后续单独做：

```text
ThreatNewsRetriever
CVEUpdateJob
SourceRanker
```

而不是直接把搜索结果喂给 LLM。

### 验收标准

```text
只允许可信来源进入报告
必须显示来源链接
搜索失败不影响主链路
LLM 不得编造搜索结果
```

## 八、P5：通知 API

### 作用

通知 API 用于高危告警通知负责人。

可选：

```text
飞书 webhook
企业微信 webhook
钉钉 webhook
SMTP 邮件
```

当前 `.env` 中还没有这些配置。

建议后续新增：

```text
NOTIFY_ENABLED=false
NOTIFY_PROVIDER=feishu
FEISHU_WEBHOOK_URL=
DINGTALK_WEBHOOK_URL=
WECHAT_WEBHOOK_URL=
SMTP_HOST=
SMTP_PORT=
SMTP_USERNAME=
SMTP_PASSWORD=
```

调用时机：

```text
critical
requires_human_review=true
business_owner 存在
automation_decision=human_review_required
```

初版建议只做 webhook，不做复杂通知编排。

## 九、总体实施顺序

建议顺序：

```text
Step 1: 完成 LLM API 接入规划中的第十四阶段
Step 2: 接 DashScope / OpenAI-compatible LLM
Step 3: 增加 fake LLM client 和测试
Step 4: 接 AbuseIPDB
Step 5: 增加威胁情报缓存
Step 6: 接 VirusTotal
Step 7: 评估是否接 Embedding + Milvus
Step 8: 评估是否接搜索 API
Step 9: 评估是否接通知 API
```

## 十、统一配置要求

所有外部 API 都必须满足：

```text
从 .env 读取配置
默认关闭
有超时时间
有失败回退
测试不调用真实 API
不能把 API Key 写入日志
不能把 API Key 提交到 GitHub
```

建议新增统一配置风格：

```text
LLM_*
THREAT_INTEL_*
ABUSEIPDB_*
VIRUSTOTAL_*
EMBEDDING_*
NOTIFY_*
```

## 十一、统一测试要求

所有外部 API 接入都要有：

```text
fake client
成功响应测试
失败响应测试
超时测试
禁用开关测试
降级策略测试
不会泄露 API Key 的测试
```

## 十二、验收标准

第十五阶段作为规划阶段，完成标准是：

```text
明确 API 接入优先级
明确每类 API 的作用
明确每类 API 的调用时机
明确每类 API 的环境变量
明确降级策略
明确测试方式
明确哪些 API 暂时不接
```

## 十三、简历描述方向

后续完成 API 接入后，可以这样描述：

```text
设计 SecRAG Agent 外部 API 接入体系，按成本和业务价值分层接入 LLM、威胁情报、Embedding、搜索和通知能力。
系统通过环境变量统一管理外部服务配置，采用按风险触发、缓存复用、超时降级和 fake client 测试机制，在提升高危告警分析质量的同时控制 token 成本、接口延迟和第三方依赖风险。
```
