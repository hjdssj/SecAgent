# 第十四阶段任务规划：LLM 报告生成与答案审校

## 一、阶段定位

第十四阶段的目标是给 SecRAG Agent 接入真正的 LLM 能力，让系统从：

```text
规则检测 + 本地 RAG 检索 + 模板化报告
```

升级为：

```text
规则检测 + RAG 检索 + 企业上下文 + LLM 证据综合 + Markdown 报告生成 + 后处理审校
```

注意：

```text
本阶段不做多 Agent。
```

原因：

```text
多 Agent 会增加 token 消耗
多 Agent 会增加响应时延
多 Agent 会增加调试复杂度
当前项目作为产品 MVP，更需要稳定、低成本、可控的 LLM 调用链路
```

本阶段只新增一个集中式 LLM 报告生成能力：

```text
SecurityAlert
  -> RAG 引用
  -> 企业上下文
  -> 威胁情报
  -> 历史 Memory
  -> PromptBuilder
  -> LLMService
  -> ReportGenerator
  -> AnswerAuditor
  -> 最终 Markdown 报告
```

## 二、当前项目真实状态

当前项目已经具备：

```text
WAF 日志接入
Redis Stream 实时事件流
攻击识别
风险评分
RAG 检索
企业上下文检索
威胁情报 mock
短期 Memory
AutoTriage 自动分流
告警持久化
前端 SOC 控制台
```

但当前还没有：

```text
LLM API 调用
LLM 生成分析报告
LLM 综合多源证据
LLM 生成自然语言处置建议
LLM 输出审校
LLM 调用成本统计
```

因此当前更准确的表述是：

```text
RAG-ready security analysis pipeline
```

完成第十四阶段后，才可以更准确地说：

```text
基于 RAG 与 LLM 的安全告警分析报告生成系统
```

## 三、LLM 在系统中的职责

LLM 不负责：

```text
第一层攻击识别
基础风险评分
WAF 规则判断
Redis 消费
数据库状态管理
是否调用高成本模块的策略判断
```

这些继续由确定性代码完成。

LLM 负责：

```text
综合证据
解释攻击判断
结合 RAG 引用生成处置建议
结合企业上下文分析业务影响
生成 Markdown 安全分析报告
对报告进行逻辑审校
把结论归纳成 SOC 人员易读的结构
```

这样做的好处：

```text
规则和模型职责清晰
成本可控
结果可解释
低风险告警不浪费 token
高风险告警获得更好的自然语言分析
```

## 四、调用策略

建议基于现有 `analysis_mode` 控制是否调用 LLM：

```text
fast
  -> 不调用 LLM
  -> 使用模板报告

enriched
  -> 默认不调用 LLM
  -> 如果 requires_human_review=true，可以调用 LLM

deep
  -> 调用 LLM 生成报告

needs_review
  -> 调用 LLM 辅助人工复核
```

推荐初版策略：

```text
analysis_mode == "deep" 时调用 LLM
requires_human_review == true 时调用 LLM
其他情况保持模板报告
```

后续可以在配置中控制：

```text
LLM_ENABLED=true
LLM_ONLY_FOR_REVIEW=true
LLM_MIN_RISK_LEVEL=high
```

## 六、环境变量规划

建议新增：

```text
LLM_ENABLED=false
LLM_PROVIDER=openai_compatible
LLM_BASE_URL=
LLM_API_KEY=
LLM_MODEL=
LLM_TIMEOUT_SECONDS=30
LLM_MAX_TOKENS=1200
LLM_TEMPERATURE=0.2
LLM_ONLY_FOR_REVIEW=true
LLM_MIN_RISK_LEVEL=high
```

说明：

```text
LLM_ENABLED - 是否启用 LLM
LLM_PROVIDER - 模型供应商类型，初版建议兼容 OpenAI 风格接口
LLM_BASE_URL - API 地址
LLM_API_KEY - API Key
LLM_MODEL - 模型名称
LLM_TIMEOUT_SECONDS - 超时时间
LLM_MAX_TOKENS - 最大输出 token
LLM_TEMPERATURE - 生成随机性，安全分析建议较低
LLM_ONLY_FOR_REVIEW - 是否只对需要人工复核的告警调用 LLM
LLM_MIN_RISK_LEVEL - 最低调用风险等级
```

## 七、可支持的模型接入方式

### 1. OpenAI-compatible API

优先建议支持 OpenAI-compatible 接口。

原因：

```text
很多模型服务都兼容这个格式
后续可以接 OpenAI、DeepSeek、通义、智谱、硅基流动、本地 vLLM 等
代码改动小
```

需要配置：

```text
LLM_BASE_URL
LLM_API_KEY
LLM_MODEL
```

### 2. 本地模型

可选支持：

```text
Ollama
LM Studio
vLLM
```

示例：

```text
LLM_BASE_URL=http://localhost:11434
LLM_MODEL=qwen2.5:7b
```

本地模型优势：

```text
不需要外部 API Key
数据不出本机
适合演示和内网部署
```

劣势：

```text
机器配置要求更高
模型效果不一定稳定
响应速度依赖硬件
```

## 八、LLM 输入设计

LLM 不应该直接读取原始日志全集。

应给 LLM 一个经过整理的结构化上下文：

```text
alert_id
event_id
attack_type
risk_score
risk_level
source_ip
target
confidence
evidence
mitre_attack
recommendations
rag_citations
threat_intel_summary
memory_summary
business_owner
asset_name
asset_criticality
context_references
triage_reason
requires_human_review
raw_log_excerpt
```

原则：

```text
给足证据
不给无限原文
引用来源保留
敏感字段可脱敏
prompt 长度可控
```

## 九、Prompt 设计

建议系统提示词约束：

```text
你是企业 SOC 安全分析助手。
你只能基于提供的证据、RAG 引用、企业上下文和威胁情报进行分析。
不要编造不存在的日志、漏洞编号、业务影响或处置动作。
如果证据不足，明确说明不确定性。
输出 Markdown。
输出必须包含结论、关键证据、攻击链分析、业务影响、处置建议、引用来源。
```

用户提示词结构：

```text
## 告警基础信息
...

## 日志证据
...

## RAG 知识库引用
...

## 企业上下文
...

## 威胁情报
...

## 历史行为
...

## 输出要求
...
```

## 十、输出格式规划

LLM 最终输出 Markdown：

```markdown
## 安全事件分析报告

### 结论

### 风险等级

### 关键证据

### 攻击链分析

### 业务影响

### 处置建议

### 引用来源

### 不确定性说明
```

要求：

```text
必须引用已提供的证据
必须保留 RAG citation
不能输出没有依据的 CVE
不能输出没有依据的封禁建议
不确定时要说明原因
```

## 十一、答案审校设计

`AnswerAuditor` 做轻量后处理，不再调用第二个 LLM。

审校内容：

```text
Markdown 是否为空
是否包含必要标题
是否包含关键证据
是否包含处置建议
是否出现明显禁用词或占位符
是否引用了不存在的来源
是否过长
```

如果审校失败：

```text
回退到模板报告
记录 llm_error
保留原始规则分析结果
```

这样可以避免 LLM 失败影响主链路。

## 十五、API 规划

本阶段不一定需要新增业务 API。

可以优先把 LLM 接入现有链路：

```text
POST /api/analyze
event_consumer
```

也就是说：

```text
生成 SecurityAlert 时按策略调用 LLM
最终 report_markdown 自动替换为 LLM 报告
```

可选新增调试 API：

```text
POST /api/alerts/{alert_id}/regenerate-report
```

作用：

```text
对已有告警重新生成 LLM 报告
```

但初版建议不做。

原因：

```text
避免扩大 API 面
避免增加状态一致性问题
先保证自动生成链路稳定
```

## 二十、暂不做

本阶段暂不做：

```text
多 Agent
自动执行封禁动作
自动调用工单系统
自动通知飞书 / 钉钉
联网搜索漏洞情报
长期对话记忆
用户侧聊天问答
流式输出
复杂 Prompt 管理平台
```

后续可以单独规划：

```text
LLM 问答助手
报告重新生成 API
人工反馈优化
Prompt 版本管理
通知联动
SOAR 动作编排
```

## 二十一、简历描述升级方向

完成后可以这样描述：

```text
设计并实现低成本 LLM 报告生成链路，在规则检测、RAG 检索、威胁情报、历史行为和企业上下文分析完成后，按风险策略选择性调用 OpenAI-compatible LLM 生成 Markdown 安全分析报告。
系统支持 fast / enriched / deep 分级调用策略、LLM 输出审校、失败回退、调用耗时与 token 元数据记录，在控制成本和时延的前提下提升高危告警的可读性与处置效率。
```
