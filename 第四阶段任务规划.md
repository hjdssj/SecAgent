# SecRAG Agent 第四阶段任务规划

## 一、阶段目标

第四阶段目标是把第三阶段的“样例日志发布到 Redis Stream”升级为“真实 WAF 日志接入”。

第三阶段已经完成：

```text
data/sample_logs/*.log
  -> publish_sample_logs.py
  -> Redis Stream: security:events
  -> event_consumer.py
  -> Redis Stream: security:alerts
  -> GET /api/alerts/recent
```

第四阶段要完成：

```text
攻击请求
  -> Nginx + ModSecurity + OWASP CRS
  -> WAF audit log
  -> waf_log_collector.py
  -> ModSecurityParser
  -> Redis Stream: security:events
  -> event_consumer.py
  -> Redis Stream: security:alerts
  -> GET /api/alerts/recent
```

这个阶段的重点是让项目真正接入 WAF 日志来源，证明系统不是只处理手写样例，而是可以消费安全设备产生的日志。

## 二、阶段价值

第四阶段完成后，项目会从：

```text
样例日志回放系统
```

升级为：

```text
WAF 驱动的实时安全分析系统
```

这一步是项目从 toy demo 变成工程化安全项目的关键。

核心价值：

```text
接入真实 WAF 日志
复用已有 Redis Stream 事件流
复用已有 Agent 分析链路
形成攻击请求到告警输出的闭环
为后续 RAG 和前端控制台提供真实事件来源
```

## 三、第四阶段核心链路

```text
curl / sqlmap / browser attack
  ↓
Nginx + ModSecurity + OWASP CRS
  ↓
WAF audit log
  ↓
backend/app/collector/waf_log_collector.py
  ↓
backend/app/collector/modsecurity_parser.py
  ↓
SecurityEvent
  ↓

  ↓
Redis Stream: security:events
  ↓
event_consumer.py
  ↓
SecurityAlert
  ↓
Redis Stream: security:alerts
```

## 四、推荐技术方案

### 1. WAF

```text
Nginx + ModSecurity v3 + OWASP Core Rule Set
```

用途：

```text
拦截 SQL 注入、XSS、路径穿越、命令注入等 Web 攻击
生成 WAF audit log
```

### 2. 靶场服务

第四阶段可以先不引入复杂靶场，使用一个简单后端服务即可。

推荐两种方案：

```text
方案 A：Nginx 直接返回静态页面
方案 B：Nginx 反向代理到一个简单 FastAPI / httpbin 服务
```

如果时间紧，先用方案 A。

### 3. 日志采集

新增：

```text
backend/app/collector/waf_log_collector.py
```

职责：

```text
持续 tail WAF audit log
读取新增日志行
调用 ModSecurityParser.parse_line()
调用 publish_event()
写入 Redis Stream: security:events
```

### 4. 消息流

继续复用第三阶段：

```text
security:events
security:alerts
security:deadletter
```

## 六、模块职责

### 1. infra/nginx/

存放 Nginx 配置。

`default.conf` 负责：

```text
监听 8080
启用 ModSecurity
配置反向代理或静态页面
指定访问日志和错误日志
```

### 2. infra/modsecurity/

存放 ModSecurity 配置。

`modsecurity.conf` 负责：

```text
启用 DetectionOnly 或 On 模式
配置 audit log
配置日志格式
```

`crs-setup.conf` 负责：

```text
配置 OWASP CRS 基础策略
```

### 3. backend/app/collector/waf_log_collector.py

负责持续采集 WAF 日志。

核心方法建议：

```text
tail_file()
collect_once()
collect_forever()
```

最小 MVP 可以先实现：

```text
collect_once(log_path)
```

### 4. scripts/simulate_attack.py

负责发送演示攻击请求。

建议支持：

```text
SQL Injection
XSS
Path Traversal
Normal Request
```

也可以先用 `curl` 命令替代脚本。

### 5. docs/waf_demo.md

记录第四阶段演示流程。

内容包括：

```text
启动 WAF
发送攻击请求
查看 WAF 日志
启动 collector
启动 consumer
查询 /api/alerts/recent
```

## 七、推荐开发顺序

### Step 1：扩展 docker-compose.yml

当前已有 Redis。

第四阶段新增 WAF 服务：

```yaml
services:
  redis:
    image: redis:7
    container_name: secrag-redis
    ports:
      - "6379:6379"

  waf:
    build:
      context: .
      dockerfile: infra/waf/Dockerfile
    container_name: secrag-waf
    ports:
      - "8080:80"
    volumes:
      - ./infra/nginx/default.conf:/etc/nginx/conf.d/default.conf
      - ./infra/modsecurity/modsecurity.conf:/etc/modsecurity/modsecurity.conf
      - ./infra/modsecurity/crs-setup.conf:/etc/modsecurity.d/owasp-crs/crs-setup.conf
      - ./data/waf_logs:/var/log/modsecurity
```

第一版可以根据实际镜像再调整路径。

### Step 2：准备 WAF Dockerfile

新增：

```text
infra/waf/Dockerfile
```

优先选择成熟镜像，避免从源码编译 ModSecurity。

推荐方向：

```text
owasp/modsecurity-crs:nginx
```

如果镜像路径或配置路径不一致，以实际容器内路径为准。

### Step 3：启动 WAF

运行：

```bash
docker compose up -d redis waf
```

检查：

```bash
docker ps
```

预期看到：

```text
secrag-redis
secrag-waf
```

访问：

```text
http://127.0.0.1:8080/
```

### Step 4：发送攻击请求

先用 curl：

```bash
curl "http://127.0.0.1:8080/login?id=1' OR '1'='1"
curl "http://127.0.0.1:8080/search?q=<script>alert(1)</script>"
curl "http://127.0.0.1:8080/download?file=../../etc/passwd"
```

预期：

```text
WAF 返回 403 或记录 audit log
```

### Step 5：确认 WAF 日志

检查本地挂载目录：

```text
data/waf_logs/
```

或进入容器：

```bash
docker exec -it secrag-waf sh
```

查看：

```text
/var/log/modsecurity/
/var/log/nginx/
```

### Step 6：编写 waf_log_collector.py

第一版可以先处理“简化日志行”或从 audit log 中提取关键字段。

建议接口：

```python
def collect_once(log_path: Path) -> int:
```

职责：

```text
读取 WAF 日志文件
解析新增行
调用 ModSecurityParser.parse_line()
调用 publish_event()
返回发布事件数量
```

如果真实 audit log 过于复杂，可以先在 collector 中抽取以下字段：

```text
client_ip
method
url
status
user_agent
rule_id
message
```

### Step 7：打通端到端链路

执行：

```bash
python scripts/simulate_attack.py
python -m app.collector.waf_log_collector
python -m app.services.event_consumer
uvicorn app.main:app --reload --port 8000
```

然后访问：

```text
GET /api/alerts/recent
```

预期返回：

```text
SQL Injection
XSS
Path Traversal
```

## 八、第四阶段验收标准

满足以下条件，即认为第四阶段完成：

```text
Redis 和 WAF 能通过 docker compose 启动
WAF 能接收 HTTP 请求
SQLi / XSS / 路径穿越请求能触发 WAF 日志
waf_log_collector.py 能读取 WAF 日志

collector 能将事件写入 security:events
event_consumer.py 能生成 security:alerts
GET /api/alerts/recent 能查询到来自 WAF 日志的告警
```

## 十、风险点

### 1. ModSecurity 镜像路径不稳定

不同镜像中的配置路径可能不同。

解决方式：

```text
先进入容器确认路径
再调整 volume 挂载
```

### 2. Audit log 格式复杂

真实 ModSecurity audit log 不是简单的一行 key=value。

解决方式：

```text
第四阶段 MVP 先解析能稳定抽取的字段
必要时先配置更易解析的日志格式
后续再扩展完整 audit log parser
```

### 3. WAF 只检测不拦截

如果 ModSecurity 是 DetectionOnly，可能不会返回 403。

解决方式：

```text
确认 SecRuleEngine On
或接受 DetectionOnly，只要 audit log 有规则命中即可
```

### 4. Windows 路径挂载问题

Docker Desktop 在 Windows 下 volume 挂载可能遇到路径或权限问题。

解决方式：

```text
优先使用项目目录下的相对路径
确认 Docker Desktop 已允许访问项目所在磁盘
```
