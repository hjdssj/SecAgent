# SecRAG Agent 第五阶段任务规划

## 一、阶段目标

第五阶段目标是把前四个阶段已经打通的“日志采集 + 攻击识别 + 告警生成”升级为“RAG 安全知识增强分析”。

第四阶段完成后，系统链路应当是：

```text
真实攻击请求
  -> WAF audit/access log
  -> waf_log_collector.py
  -> Redis Stream: security:events
  -> event_consumer.py
  -> SecurityAlert
```

第五阶段要新增：

```text
ParsedSecurityEvent / SecurityAlert
  -> RAG 检索 Agent
  -> 安全知识库检索
  -> CVE / ATT&CK / OWASP CRS / 修复建议
  -> 带引用的 Markdown 分析报告
```

本阶段重点不是追求超大规模知识库，而是先建立一个可演示、可解释、可引用的本地安全知识库。

## 二、阶段价值

第五阶段完成后，项目不再只是基于规则判断攻击类型，而是可以结合安全知识回答：

```text
这个攻击属于哪类攻击手法？
命中了哪些 WAF/CRS 规则？
对应 ATT&CK 哪些技术？
可能关联哪些 CVE 或漏洞类型？
应该如何修复？
答案依据来自哪些知识条目？
```

这一步是 SecRAG Agent 中 “RAG” 的核心体现。

## 三、核心链路

```text
SecurityAlert
  -> query_rewriter.py
  -> RAGAgent
  -> LlamaIndex
  -> BM25 + Vector Search
  -> knowledge_base/*
  -> evidence cards
  -> report_markdown
```

## 四、推荐知识库范围

MVP 先准备小而清晰的知识库：

```text
SQL Injection
XSS
Path Traversal
Command Injection
Brute Force
Automated Scanner
OWASP CRS 规则说明
MITRE ATT&CK 技术映射
常见修复建议
```

如果时间充足，再加入：

```text
CVE 示例数据
真实漏洞公告摘要
Nginx / WAF 加固建议
```

## 六、模块职责

### 1. knowledge_loader.py

负责读取本地 Markdown 知识库，并交给 LlamaIndex 建索引。

建议先支持：

```text
读取 backend/app/data/knowledge_base/*.md
构建 VectorStoreIndex
持久化到 data/index_storage/
```

### 2. query_rewriter.py

负责把安全事件改写成适合检索的查询。

示例：

```text
原始事件：
attack_type=SQL Injection
waf_rule_id=942100
url=/login?id=1' OR '1'='1

改写查询：
SQL Injection authentication bypass OWASP CRS 942100 MITRE remediation parameterized query
```

### 3. rag_agent.py

负责执行检索并返回结构化检索结果。

建议输出：

```text
summary
references
recommended_actions
matched_knowledge
```

### 4. schemas.py

定义 RAG 检索结果模型，例如：

```text
KnowledgeReference
RAGAnalysisResult
```

### 5. orchestrator.py

在现有分析链路中加入 RAG：

```text
LogParserAgent
  -> DecisionAgent
  -> RAGAgent
  -> final SecurityAlert
```

## 七、推荐开发顺序

### Step 1：准备知识库 Markdown

先写 4 到 5 个小型知识文件，不需要很长，但要结构清晰。

每个知识条目建议包含：

```text
攻击类型
典型特征
WAF/CRS 规则
ATT&CK 映射
修复建议
引用标题
```

### Step 2：新增 RAG 数据模型

文件：

```text
backend/app/rag/schemas.py
```

先定义最小模型：

```text
KnowledgeReference
RAGAnalysisResult
```

### Step 3：编写 Query Rewrite

文件：

```text
backend/app/rag/query_rewriter.py
```

输入：

```text
ParsedSecurityEvent 或 SecurityAlert
```

输出：

```text
适合安全知识检索的 query 字符串
```

### Step 4：编写 RAGAgent

文件：

```text
backend/app/rag/rag_agent.py
```

第一版可以先不用复杂向量库，先用 LlamaIndex 本地索引或轻量检索跑通。

### Step 5：接入 Orchestrator

在生成最终 `SecurityAlert` 前，追加 RAG 检索结果。

重点修改：

```text
evidence 增加知识库证据
recommendations 增加 RAG 修复建议
report_markdown 增加引用区
```

### Step 6：测试三类攻击

至少测试：

```text
SQL Injection
XSS
Path Traversal
```

确认报告中出现：

```text
攻击解释
ATT&CK 映射
修复建议
知识库引用
```

## 八、验收标准

满足以下条件，即认为第五阶段完成：

```text
本地知识库 Markdown 文件存在
RAGAgent 可以根据攻击事件检索知识
SQLi / XSS / Path Traversal 都能检索到相关知识
SecurityAlert 中能包含 RAG 增强证据
report_markdown 中能显示引用来源
系统在没有外部 API 的情况下也能完成知识增强分析
```

## 九、暂不实现

第五阶段先不做：

```text
大规模 CVE 全量同步
Milvus 集群部署
复杂 rerank 模型
联网实时检索
多轮问答式 RAG
前端知识库管理页面
```

这些放到后续增强阶段。

## 十、风险点

### 1. 依赖安装过重

LlamaIndex 依赖较多，建议先锁定最小依赖，避免一次性安装太多组件。

### 2. 中文编码问题

知识库 Markdown 使用 UTF-8 保存，避免 PowerShell 显示乱码影响判断。

### 3. 检索结果不稳定

MVP 可以先用较小知识库保证命中稳定，再逐步扩展内容。
