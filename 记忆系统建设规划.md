# 记忆系统建设规划

## 目标

把当前 SecAgent 从“单次告警分析系统”升级为“可追问、可复盘、可长期学习的安全分析系统”。

本规划不引入多 Agent，不让 LLM 接管主判定。记忆系统的职责是保存上下文、复用分析经验、支持追问和人工管理，核心攻击识别、风险评分和处置流仍由现有规则、RAG、威胁情报、AutoTriage 和可选 LLM 报告增强完成。

## 当前已有能力

当前代码中已经有一层 Redis 短期历史记忆：

```text
backend/app/memory/event_memory.py
```

现有能力：

```text
按 source_ip 记录历史告警数量
记录该 IP 历史攻击类型
记录该 IP 历史目标路径
保存最近 50 条该 IP 告警 JSON
在 enriched / deep 模式中读取历史行为并调整风险分
```

当前 Redis key 形态：

```text
memory:ip:{source_ip}
memory:ip:{source_ip}:attack_types
memory:ip:{source_ip}:targets
memory:ip:{source_ip}:alerts
```

当前还没有：

```text
一次分析内的显式 state 对象
会话级 findings / workflow / reflections
跨会话长期学习向量库
/memories 系列管理接口
分析结束后的追问接口
security:result:{session_id} 问答上下文缓存
```

## 总体分层设计

### 1. 一次分析内：State 共享上下文

作用：

```text
在一次 analyze 调用中保存所有中间结果
避免模块之间只靠 SecurityAlert 传递信息
为后续追问沉淀标准上下文
```

建议新增：

```text
backend/app/analysis/state.py
```

核心对象：

```text
AnalysisState
```

建议字段：

```text
session_id
event
parsed_event
llm_unknown_classification
base_alert
rag_result
context_result
triage_result
threat_intel_result
memory_summary
final_alert
findings
workflow_steps
reflections
created_at
```

落地方式：

```text
第一步只在 SecurityAnalysisOrchestrator.analyze() 内部使用内存对象
不改变外部 API 返回结构
确认稳定后再把 state 摘要写入 Redis
```

## 2. 当前会话内：Redis 存 findings / workflow / reflections

作用：

```text
保存一次分析的过程信息
支持前端展示分析链路
支持分析结束后的追问
支持短期会话恢复
```

建议 Redis key：

```text
security:session:{session_id}:state
security:session:{session_id}:findings
security:session:{session_id}:workflow
security:session:{session_id}:reflections
security:session:{session_id}:messages
```

建议 TTL：

```text
默认 24 小时
可配置 SESSION_MEMORY_TTL_SECONDS=86400
```

findings 示例：

```json
{
  "type": "attack_detection",
  "title": "检测到 SSRF",
  "description": "LLM Unknown fallback 识别到元数据地址访问行为",
  "evidence": ["169.254.169.254", "url parameter controls destination"],
  "confidence": 0.92,
  "source": "llm_unknown_classifier"
}
```

workflow 示例：

```json
{
  "step": "rag_retrieval",
  "status": "completed",
  "summary": "检索到 SSRF 与云元数据访问相关知识",
  "latency_ms": 12.5
}
```

reflections 示例：

```json
{
  "summary": "本次事件命中规则较少，但 payload 指向云元数据 IP，后续应补充 SSRF 规则。",
  "suggested_knowledge_update": true,
  "suggested_rule_update": true
}
```

## 3. 分析结束追问：Redis security:result:{session_id}

作用：

```text
用户在分析结束后继续问：
为什么判高危？
证据有哪些？
为什么需要人工复核？
这个 IP 历史上还打过什么？
应该怎么处置？
```

建议 Redis key：

```text
security:result:{session_id}
```

保存内容：

```text
final_alert
parsed_event
score_breakdown
analysis_metadata
rag_references
context_references
threat_intel_result
memory_summary
workflow_steps
findings
reflections
```

建议新增 API：

```text
POST /api/analysis/{session_id}/ask
GET /api/analysis/{session_id}/result
```

说明：

```text
ask 接口需要 LLM API
LLM 只基于 security:result:{session_id} 中的上下文回答
不重新做攻击判定
不改变原始告警状态
```

## 4. 跨会话长期学习：Milvus 存安全分析向量和摘要

作用：

```text
沉淀历史分析经验
让相似攻击、相似资产、相似处置动作可以被后续检索
支持“这个事件和历史哪个事件像？”
支持“类似事件以前怎么处理？”
```

建议 Milvus collection：

```text
secagent_analysis_memory
```

建议字段：

```text
memory_id
alert_id
session_id
source_ip
target
attack_type
risk_level
business_owner
asset_criticality
summary
evidence_text
recommendation_text
resolution_status
analyst_note
created_at
embedding
```

写入时机：

```text
告警生成后，可以先写入候选长期记忆
人工处理完成后，更新 resolution_status 和 analyst_note
只有 resolved / false_positive / high confidence 的事件进入长期学习更稳妥
```

检索时机：

```text
enriched / deep 分析模式中检索长期记忆
追问接口中检索长期记忆
前端记忆页面中按条件检索长期记忆
```

注意：

```text
当前项目还没有真正接入 Milvus
可以先实现接口层和本地 fallback
等你确认 Milvus 部署方式后再接真实向量库
```

## 存储边界

### Redis 存什么

```text
实时事件流 security:events
实时告警流 security:alerts
deadletter
IP 短期历史记忆 memory:ip:{source_ip}
会话 state / findings / workflow / reflections
分析结果追问上下文 security:result:{session_id}
短 TTL 缓存
```

特点：

```text
快
适合短期上下文
适合流式消费
可以设置 TTL
不适合长期审计和复杂筛选
```

### 数据库存什么

```text
告警主记录
告警状态
人工备注
处理人
处理时间
LLM 元数据
可审计的业务闭环数据
```

特点：

```text
稳定
可分页查询
可做审计
适合 SOC 生命周期闭环
```

### Milvus 存什么

```text
长期安全经验摘要
历史事件向量
处置经验向量
相似攻击检索索引
```

特点：

```text
适合语义相似检索
不适合做唯一事实源
不替代数据库
```

## 开发阶段安排

### 第一阶段：AnalysisState 内存上下文

目标：

```text
新增 AnalysisState
改造 Orchestrator 内部链路
不改变 API 返回
不引入 Redis / Milvus 新依赖
```

验收：

```text
现有测试全部通过
新增测试验证 state 中包含 parsed_event、final_alert、workflow_steps
```

### 第二阶段：Redis 会话记忆

目标：

```text
新增 SessionMemoryStore
分析结束后写入 security:result:{session_id}
写入 findings / workflow / reflections
支持 TTL
```

验收：

```text
Redis 可看到 security:result:{session_id}
consumer 分析后能写入会话结果
Redis 不可用时主分析链路不失败
```

### 第三阶段：追问接口

目标：

```text
新增 GET /api/analysis/{session_id}/result
新增 POST /api/analysis/{session_id}/ask
ask 使用 LLM 基于 session result 回答
```

验收：

```text
能追问“为什么是 high”
能追问“证据有哪些”
能追问“下一步怎么处理”
LLM 失败时返回明确错误，不影响原告警
```

### 第四阶段：长期记忆抽象层

目标：

```text
新增 LongTermMemoryStore 接口
先实现 InMemory / local JSON fallback
定义 MilvusMemoryStore 结构
```

验收：

```text
不部署 Milvus 也能跑测试
接口层稳定
后续可替换为真实 Milvus
```

### 第五阶段：Milvus 接入

目标：

```text
接入 pymilvus
写入安全分析摘要向量
支持相似历史事件检索
```

需要配置：

```text
MILVUS_HOST
MILVUS_PORT
MILVUS_COLLECTION
EMBEDDING_PROVIDER
EMBEDDING_MODEL
EMBEDDING_API_KEY
```

验收：

```text
分析后可写入长期记忆
相似事件能被检索回来
Milvus 不可用时降级，不中断主链路
```

### 第六阶段：/memories 管理接口与前端页面

目标：

```text
新增 memories API
新增前端 Memories 页面
支持查询、详情、禁用、手动新增
```

验收：

```text
前端能看到长期记忆列表
能禁用错误记忆
禁用后的记忆不再参与检索增强
```

## 需要的外部 API / 服务

必须：

```text
Redis
LLM API：用于追问回答和可选 reflection 摘要
```

长期记忆阶段需要：

```text
Milvus
Embedding API 或本地 embedding 模型
```

可选：

```text
对象存储：如果未来支持上传大量原始文档或附件
```

## 风险与约束

```text
不能把 Redis 当长期数据库
不能把 Milvus 当事实数据库
LLM 追问只能基于已有上下文回答，不能重写告警事实
长期记忆需要人工禁用/纠错机制，否则错误经验会被反复复用
Memory 检索结果必须在报告中标注来源，避免黑箱建议
```

## 最终目标链路

```text
WAF 日志
  -> SecurityEvent
  -> AnalysisState
  -> 规则解析 / Unknown LLM fallback / RAG / ThreatIntel / Memory / AutoTriage
  -> SecurityAlert
  -> DB alerts
  -> Redis security:result:{session_id}
  -> 可追问问答
  -> 人工处理闭环
  -> 高质量摘要进入 Milvus 长期记忆
  -> 后续相似事件检索复用
```
