# 第十阶段任务规划：成本可控的安全分析流水线与可解释风险评分

## 一、阶段定位

第十阶段不继续强化“多 Agent”概念，而是从产品落地角度优化系统。

核心目标是让 SecRAG Agent 更像一个真实 SOC 产品：

```text
低成本
低时延
可解释
可批量处理
能按风险分层分析
能让安全运营人员看懂为什么告警
```

本阶段的关键词是：

```text
规则优先
RAG 按需
情报按需
缓存复用
评分可解释
分析链路可观测
```

## 二、为什么要做这一阶段

真实企业安全场景中，日志和告警量很大。

如果每条日志都走复杂多 Agent 或大模型推理，会产生几个问题：

```text
token 成本高
接口时延高
吞吐能力差
链路不稳定
故障排查复杂
安全运营人员难以信任黑盒结论
```

所以本项目应该采用更适合产品化的设计：

```text
先用规则和 WAF 信息快速判断
只对可疑或高危事件触发 RAG
只对中高危事件触发威胁情报
只在需要报告时生成 Markdown 分析
所有风险分数都要能解释
```

## 三、阶段目标

第十阶段完成后，系统应该支持：

```text
analysis_mode 分析模式
RiskScoreBreakdown 风险评分拆解
按风险等级决定是否触发 RAG
按风险等级决定是否查威胁情报
记录每次分析耗时
记录启用了哪些增强模块
告警报告展示评分原因
测试覆盖不同风险路径
```

最终效果：

```text
普通请求走 fast 模式，快速返回
中风险请求走 enriched 模式，加入 RAG / Memory
高危请求走 deep 模式，加入 RAG / ThreatIntel / Memory / 完整报告
```

## 四、推荐分析模式

### fast

适用：

```text
低风险或正常请求
```

启用模块：

```text
LogParserAgent
DecisionAgent
```

特点：

```text
成本最低
延迟最低
不触发 RAG
不查威胁情报
只输出基础判断
```

### enriched

适用：

```text
中风险或命中可疑攻击特征的请求
```

启用模块：

```text
LogParserAgent
DecisionAgent
RAGAgent
EventMemory
```

特点：

```text
补充知识库引用
补充历史行为
不一定查外部威胁情报
适合大部分 SOC 告警
```

### deep

适用：

```text
高危或 critical 事件
命中 WAF 高危规则
源 IP 有历史攻击记录
攻击 payload 明确
```

启用模块：

```text
LogParserAgent
DecisionAgent
RAGAgent
ThreatIntelAgent
EventMemory
Markdown 报告增强
```

特点：

```text
分析最完整
成本最高
只用于高价值告警
```

## 五、建议新增数据结构

### AnalysisMode

建议定义：

```python
AnalysisMode = Literal["fast", "enriched", "deep"]
```

### RiskScoreItem

字段建议：

```text
name: 评分项名称
score: 本项加分或减分
reason: 加减分原因
source: 评分来源，例如 rule / waf / rag / intel / memory
```

示例：

```text
WAF 命中 SQLi 规则 +25
SQL 注入 payload 特征 +30
sqlmap User-Agent +15
RAG 命中 SQLi 知识 +8
威胁情报高危 IP +20
历史行为命中 +10
```

### RiskScoreBreakdown

字段建议：

```text
base_score
items
total_score
risk_level
confidence
```

### AnalysisMetadata

字段建议：

```text
analysis_mode
enabled_modules
skipped_modules
latency_ms
rag_used
threat_intel_used
memory_used
```

## 七、推荐实现链路

### 1. 初步解析

```text
SecurityEvent
  -> LogParserAgent
  -> ParsedSecurityEvent
```

### 2. 基础决策

```text
ParsedSecurityEvent
  -> DecisionAgent
  -> base SecurityAlert
  -> RiskScoreBreakdown 初始项
```

基础评分来自：

```text
attack_type
attack_features
waf_rule_id
waf_message
confidence
status
user_agent
```

### 3. 分析模式选择

```text
base alert
  -> AnalysisPolicy
  -> fast / enriched / deep
```

建议规则：

```text
risk_score < 40 -> fast
40 <= risk_score < 80 -> enriched
risk_score >= 80 -> deep
critical -> deep
```

### 4. 按模式触发增强模块

fast：

```text
不触发 RAG
不触发 ThreatIntel
不触发 Memory 或只做轻量读取
```

enriched：

```text
触发 RAG
触发 Memory
不触发 ThreatIntel 或只查缓存
```

deep：

```text
触发 RAG
触发 ThreatIntel
触发 Memory
生成完整 Markdown 报告
```

### 5. 输出最终告警

最终 `SecurityAlert` 建议增加：

```text
analysis_mode
score_breakdown
analysis_metadata
```

前端可以展示：

```text
为什么是 high
哪些模块参与了分析
哪些模块被跳过
本次分析耗时多少
```

## 十二、简历描述升级方向

完成第十阶段后，可以这样描述：

```text
针对 SOC 场景下日志量大、时延敏感和成本敏感的问题，设计成本可控的安全分析流水线。
系统先通过规则和 WAF 命中信息完成快速研判，再根据风险等级动态选择 fast / enriched / deep 分析模式；
仅对中高危事件触发 RAG、威胁情报和历史行为分析，并输出 RiskScoreBreakdown、启用模块、跳过模块和分析耗时。
该设计降低了不必要的大模型和外部情报调用成本，同时提升告警结论的可解释性和产品落地性。
```

## 十三、暂不做

本阶段暂不做：

```text
多 Agent 自主协作
每条日志都调用 LLM
复杂工作流引擎
真实用户权限系统
完整工单系统
外部威胁情报真实 API 接入
```

这些可以作为后续产品增强阶段。
