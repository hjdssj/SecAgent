# 第十三阶段任务规划：单服务器可部署 WAF 接入版

## 一、阶段定位

第十三阶段的目标不是继续增加分析算法，而是把当前已经跑通的 SecRAG Agent 从“本地演示链路”升级为“可以部署在一台服务器上保护真实业务的 WAF 接入方案”。

当前系统已经具备：

```text
WAF 日志解析
Redis Stream 实时事件流
攻击识别与风险评分
真实 RAG 检索
企业上下文 RAG
AutoTriage 自动分流
告警持久化
前端 SOC 控制台
```

但目前 WAF 仍然偏演示配置，主要用于本地生成和采集日志。

本阶段要规划的是：

```text
把 WAF 放到真实业务服务前面
让 WAF 作为反向代理接收外部请求
让 ModSecurity / OWASP CRS 生成真实 audit log
让 collector 持续采集 audit log
让 consumer 持续分析并入库
让前端 SOC 控制台查看真实告警
```

目标链路：

```text
用户请求
  -> 服务器公网 IP / 域名
  -> Nginx + ModSecurity + OWASP CRS WAF
  -> 后端真实业务服务
  -> WAF audit.log
  -> waf_log_collector.py
  -> Redis Stream: security:events
  -> event_consumer.py
  -> SecurityAnalysisOrchestrator
  -> Redis Stream: security:alerts
  -> DB alerts 表
  -> FastAPI
  -> React SOC Console
```

## 二、本阶段要解决的问题

### 1. WAF 如何代理真实业务

当前 WAF 只是返回：

```text
SecRAG WAF is running
```

真实部署时需要把 Nginx 配成反向代理：

```text
location /
  -> proxy_pass http://真实业务服务地址
```

例如：

```text
http://127.0.0.1:3000
http://backend:8080
http://10.0.0.12:8080
```

### 2. WAF 规则模式如何选择

ModSecurity 常见模式：

```text
DetectionOnly - 只检测不拦截
On - 检测并拦截
Off - 关闭规则
```

建议部署初期使用：

```text
DetectionOnly
```

原因：

```text
先观察误报
先验证日志链路
先让 SecAgent 产生告警
避免一上来误拦截真实业务请求
```

稳定后再切换：

```text
On
```

### 3. audit log 如何持久化

WAF 容器内部日志路径：

```text
/var/log/modsecurity/audit/audit.log
```

宿主机挂载路径：

```text
data/waf_logs/modsecurity/audit/audit.log
```

部署版需要保证：

```text
容器重启后日志不丢
collector 可以持续读取
日志目录权限正确
日志文件增长可控
```

### 4. collector 如何常驻

当前 collector 可以运行：

```powershell
cd backend
python -m app.collector.waf_log_collector --follow
```

部署版需要规划成常驻服务：

```text
Docker Compose service
或 systemd service
或 supervisor 进程
```

建议优先使用：

```text
Docker Compose service
```

### 5. consumer 如何常驻

当前 consumer 可以单次消费：

```powershell
cd backend
python -m app.services.event_consumer
```

部署版需要升级为持续消费模式。

规划方向：

```text
新增 --follow 参数
循环调用 consume_once()
支持消费间隔
支持异常捕获
支持失败事件写入 deadletter
```

### 6. backend / frontend 如何部署

后端：

```text
FastAPI + uvicorn
```

前端：

```text
React build 后由 Nginx 静态托管
或继续使用独立 Vite preview
```

单服务器优先建议：

```text
backend 单独容器
frontend build 后由 Nginx 或独立前端容器托管
```

## 四、部署版 Docker Compose 规划

建议单服务器部署版包含：

```text
redis
waf
backend
frontend
collector
consumer
```

职责：

```text
redis - 实时事件流、短期记忆、缓存
waf - Nginx + ModSecurity + OWASP CRS，代理真实业务
backend - FastAPI API 服务
frontend - SOC 控制台
collector - 持续采集 WAF audit.log 并写入 Redis
consumer - 持续消费 Redis 事件并生成告警
```

部署链路：

```text
waf
  -> 写 data/waf_logs/modsecurity/audit/audit.log
collector
  -> 读 audit.log
  -> 写 Redis security:events
consumer
  -> 读 Redis security:events
  -> 写 Redis security:alerts
  -> 写 DB alerts 表
backend
  -> 查询 DB alerts 表
frontend
  -> 调用 backend API
```

## 十、风险与注意事项

### 1. 误报风险

OWASP CRS 在真实业务中可能误报。

建议：

```text
先 DetectionOnly
观察 1 到 3 天
记录误报接口
再逐步开启 On
```

### 2. 日志膨胀

WAF audit.log 会持续增长。

需要后续补：

```text
logrotate
日志保留周期
磁盘占用监控
```

### 3. HTTPS 复杂度

如果要让 WAF 接公网域名，通常需要 HTTPS。

可选方案：

```text
手动挂载证书
Nginx 配置 443
后续接 Let's Encrypt
```

### 4. 性能影响

ModSecurity 会增加请求处理开销。

建议：

```text
先保护关键接口
压测后再扩大保护范围
```
