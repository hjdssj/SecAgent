# 工作日志

## 第一阶段目标：后端单条日志分析闭环

### 目标说明

第一阶段先不接入前端、WAF、Redis、Milvus、LlamaIndex 和真实威胁情报。

当前目标是完成一个最小可运行闭环：

```text
输入一条 WAF 日志 JSON
  -> 标准化为 SecurityEvent
  -> LogParserAgent 解析攻击特征
  -> DecisionAgent 判断攻击类型和风险等级
  -> FastAPI 返回 SecurityAlert 告警 JSON
```

这个阶段的重点是先把后端分析链路跑通，为后续接入 WAF 实时日志、RAG 知识库、威胁情报和前端控制台打基础。

### 开发顺序

1. 定义 `SecurityEvent`，明确原始安全事件的数据结构。
2. 定义 `ParsedSecurityEvent`，表示日志解析后的中间结果。
3. 定义 `SecurityAlert`，明确最终告警返回格式。
4. 编写 `LogParserAgent`，使用规则识别 SQL 注入、XSS、路径穿越、命令注入和扫描器特征。
5. 编写 `DecisionAgent`，根据攻击类型、WAF 规则、状态码和攻击特征计算风险分数。
6. 编写 `SecurityAnalysisOrchestrator`，串联解析和决策流程。
7. 编写 `POST /api/analyze` 接口，输入 `SecurityEvent`，输出 `SecurityAlert`。
8. 使用 SQL 注入样例日志进行接口测试。


### 调试记录

本阶段遇到并解决了以下问题：

1. `LogParserAgent` 继承 `BaseModel` 导致 Pydantic 将规则列表识别为模型字段。

问题表现：

```text
PydanticUserError: A non-annotated attribute was detected: SQLI_PATTERNS
```

解决方式：

```text
LogParserAgent 是业务逻辑类，不是数据模型，因此不继承 BaseModel，改为普通 class。
```

2. `DecisionAgent` 方法缩进需要确保位于类内部。

问题风险：

```text
如果 decide() 等方法缩进到类外部，会导致 DecisionAgent 实例没有 decide 方法。
```

解决方式：

```text
保证 decide()、_score()、_level()、_map_mitre()、_recommendations()、_build_report() 都在 DecisionAgent 类内部。
```

3. 中文注释曾出现乱码。

后续注意：

```text
Python 文件统一使用 UTF-8 编码保存。
```
