### 第一阶段调试记录

1. `LogParserAgent` 最初继承 `BaseModel`，导致 Pydantic 将规则列表识别为模型字段。

解决方式：

```text
LogParserAgent 是业务逻辑类，不是数据模型，因此改为普通 class。
```

2. `DecisionAgent` 方法需要确保位于类内部。

解决方式：

```text
保证 decide()、_score()、_level()、_map_mitre()、_recommendations()、_build_report() 都缩进在 DecisionAgent 内部。
```

3. 中文注释曾出现编码异常。

解决方式：

```text
Python 文件和 Markdown 文件统一使用 UTF-8 保存。
```

## 第十阶段：成本可控的安全分析流水线与可解释评分

### 阶段目标

核心思路：

```text
规则优先
RAG 按需

评分可解释
分析耗时可观测
```

模块职责：

### 分析模式设计

已实现三种分析模式：

```text
fast

deep
```

模式策略：

```text
fast - 低风险事件，仅执行日志解析和基础决策，跳过 RAG、ThreatIntel、Memory
enriched - 中风险事件，执行 RAG 和 Memory，跳过 ThreatIntel
deep - 高危事件，执行 RAG、ThreatIntel、Memory 全量增强
```

## LLM 增强：Unknown 攻击类型补充识别

### 完成时间

2026-06-07

### 背景

规则引擎和 WAF 规则可以覆盖常见攻击，但真实日志中仍然会出现规则未命中的请求。

如果所有无法识别的事件都直接归为 `Unknown`，会造成：

```text
风险分数偏低
处置建议过于泛化
RAG 检索上下文不够准确
SOC 分析人员需要手动判断攻击类型
```

### 设计原则

本次没有引入多 Agent，也没有让 LLM 接管主判断。

LLM 只作为 `Unknown` 专用 fallback：

```text
LogParserAgent 规则识别成功 -> 不调用 LLM
LogParserAgent 输出 Unknown -> 根据配置决定是否调用 LLM
LLM 只返回攻击类型、置信度、理由和命中特征
只有高置信度且攻击类型受支持时才采纳
低置信度、失败、不可识别类型均保留 Unknown
```

### 新增配置

```text
LLM_UNKNOWN_CLASSIFIER_ENABLED=false
LLM_UNKNOWN_CLASSIFIER_MAX_TOKENS=500
LLM_UNKNOWN_CLASSIFIER_MIN_CONFIDENCE=0.7
```

说明：


### 背景

项目已经具备 `AnalysisState` 短期会话记忆和 `POST /api/analysis/{session_id}/ask` 追问接口，但前端告警详情页还没有可用的多轮对话入口。
本次补充的是告警详情内的 Follow-up 对话框，让分析人员可以围绕某个已保存 `session_id` 的告警继续追问。

### 已完成内容

```text
AnalysisFollowupRequest 新增 history 字段
新增 AnalysisFollowupMessage，表示当前前端对话中的 user / assistant 历史消息
AnalysisFollowupAnswerer 支持把 history 写入 prompt
POST /api/analysis/{session_id}/ask 支持接收多轮历史
前端新增 frontend/src/types/analysis.ts
前端新增 frontend/src/api/analysis.ts
前端新增 FollowupChat 组件
AlertDetail 中接入 FollowupChat
FollowupChat 支持问题输入、答案展示、本地多轮历史、错误展示和 loading 状态
切换告警时自动清空当前对话历史，避免不同告警上下文混用
前端样式新增 followup-panel / followup-thread / followup-bubble / followup-form
```

### 能力边界

```text
Follow-up 回答仍然只基于 Redis 中保存的 AnalysisState 和当前前端传入的对话历史
Follow-up 不会重新执行攻击识别
Follow-up 不会修改 risk_score、risk_level、alert status 或数据库记录
没有 session_id 的历史告警无法追问
当前对话历史保存在前端组件状态中，刷新页面后不会保留
```
