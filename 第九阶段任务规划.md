# 第九阶段任务规划：真实 RAG 升级

## 一、阶段目标

第九阶段的核心目标是把当前偏 MVP 的本地安全知识检索能力，升级为更接近真实项目的 RAG 检索链路。

当前系统已经具备：

```text
WAF 日志接入
Redis Stream 事件流
LogParserAgent 攻击识别
RAGAgent 本地知识增强
ThreatIntelAgent 威胁情报
EventMemory 历史行为记忆
React SOC 告警展示
```

但是 RAG 部分目前还比较轻量，更像基于本地 Markdown 的关键词检索。第九阶段需要补强：

```text
安全知识文档结构化
知识切分
索引构建
BM25 检索
向量检索预留
混合召回
Query Rewrite
检索证据引用
检索评分解释
```

最终目标是让项目在被追问 “RAG 到底怎么做的” 时，可以讲清楚数据、索引、召回、排序、引用和增强生成链路。

## 二、为什么先做 RAG 升级

项目名称是 SecRAG Agent，RAG 是核心亮点。

如果 RAG 只是简单读取几个 Markdown 文件，会比较容易被认为是风味功能。第九阶段完成后，RAG 能从简历描述中的“概念词”变成真实工程模块：

```text
原始安全知识
  -> 文档加载
  -> 文档切分
  -> 元数据标注
  -> 索引构建
  -> Query Rewrite
  -> BM25 / 向量召回
  -> 混合排序
  -> 引用片段返回
  -> 告警证据增强
```

这样可以明显增强项目可信度。

## 三、阶段范围

### 本阶段要做

第九阶段建议完成以下内容：

```text

10. 告警 evidence 和 report_markdown 增加引用来源
11. 增加 RAG 检索测试用例
12. 更新 README、工作日志和架构文档
```

### 本阶段暂不做

为了控制两天内的实现复杂度，本阶段暂不强行接入完整生产级向量库。

暂不做：

```text

大规模 CVE 数据同步
LLM 在线总结
复杂 reranker
多租户知识库
知识库后台管理页面
```

但是代码结构要预留这些扩展点。

## 五、数据模型设计

### KnowledgeDocument

用于表示一篇原始知识文档。

字段建议：

```text
doc_id: 文档唯一 ID
title: 文档标题
category: 文档类别，例如 attack / mitre / cve / remediation / waf_rule
source: 来源文件或来源名称
tags: 标签，例如 sqli、xss、T1190
content: 文档正文
```

### KnowledgeChunk

用于表示切分后的检索片段。

字段建议：

```text
chunk_id: 片段唯一 ID
doc_id: 所属文档 ID
title: 文档标题
category: 文档类别
source: 来源文件
tags: 标签
content: 片段正文
score: 检索得分
```

### RetrievalResult

用于表示最终返回给 RAGAgent 的检索结果。

字段建议：

```text
chunk_id
title
source
category
tags
content
score
retrieval_type: bm25 / vector / hybrid
reason: 命中原因说明
```

## 六、检索链路设计

### 1. 文档加载

读取 `backend/app/data/knowledge/` 下的 Markdown 文件。

每个文件可以用简单 front matter 或文件名推断元数据。

示例：

```markdown
# SQL Injection

category: attack
tags: sqli, sql injection, 942100, T1190

SQL 注入是一类通过构造恶意 SQL 片段影响数据库查询逻辑的攻击。
```

### 2. 文档切分

先采用简单可靠的切分策略：

```text
按标题切分
按段落切分
限制 chunk 最大字符数
保留标题和标签
```

不建议一开始做过度复杂的 semantic chunk。

### 3. Query Rewrite

输入来自 ParsedSecurityEvent。

例如：

```text
attack_type = SQL Injection
waf_rule_id = 942100
url = /login?id=1' OR '1'='1
user_agent = sqlmap/1.7
```

改写为：

```text
SQL Injection sqlmap waf rule 942100 login parameter injection T1190 remediation
```

这样可以提升检索命中率。

### 4. BM25 检索

实现一个轻量 BM25 检索器。

输入：

```text
query
top_k
optional filters
```

输出：

```text
score 排序后的 KnowledgeChunk 列表
```

BM25 可以先自己实现一个轻量版，也可以引入成熟库。为了简历可讲性，建议先写轻量实现。

### 5. 向量检索预留

本阶段可以先写接口，不一定真实接入 Milvus。

示例：

```text
VectorRetriever.search(query, top_k)
```

当前返回空列表或使用简单 fallback。

后续可以替换为：

```text
BGE embedding
Milvus collection
向量 top_k 检索
```

### 6. 混合召回

HybridRetriever 负责融合：

```text
BM25 结果
Vector 结果
规则增强结果
```

排序策略可以先用：

```text
hybrid_score = bm25_score * 0.7 + vector_score * 0.3
```

如果本阶段 VectorRetriever 还没真实实现，则 hybrid 仍然返回 BM25 结果，但结构上保留 vector 扩展能力。

## 七、RAGAgent 改造

当前 RAGAgent 应该从简单本地匹配升级为：

```text
ParsedSecurityEvent
  -> QueryRewriter
  -> HybridRetriever
  -> RetrievalResult
  -> RAGContext
  -> 写入 SecurityAlert evidence / recommendations / report_markdown
```

RAGAgent 输出建议包含：

```text
matched_knowledge: 检索片段列表
evidence: 可读证据
recommendations: 处置建议
citations: 引用来源
```

告警里的证据可以变成：

```text
RAG 引用：SQL Injection - attack_sqli.md，score=3.82
RAG 引用：OWASP CRS 942100 - waf_rules.md，score=2.91
```

报告中可以增加：

```markdown
**知识库引用**

- SQL Injection（attack_sqli.md）
- OWASP CRS 942100（waf_rules.md）
```

## 十一、简历描述升级方向

完成第九阶段后，可以把原来的描述升级为：

```text
基于结构化安全知识库构建 RAG 检索链路，设计 KnowledgeDocument / KnowledgeChunk 数据模型，
实现文档加载、段落切分、Query Rewrite、BM25 召回与 HybridRetriever 混合检索框架。
在安全告警分析过程中，根据攻击类型、WAF 规则、URL 特征与 User-Agent 自动改写检索查询，
召回 SQL 注入、XSS、路径穿越、OWASP CRS、MITRE ATT&CK 与修复建议等知识片段，
并在最终告警报告中输出引用来源、命中分数和处置建议，提高分析结果的可解释性。
```

## 十二、阶段风险

主要风险：

```text
RAG 模块文件较多，容易写散
知识文档质量会影响检索效果
BM25 分词如果过于粗糙，中文和英文混合检索效果可能一般
向量检索如果强行接入 Milvus，会占用太多时间
```

规避方式：

```text
先做轻量但完整的 RAG 链路
优先保证 source / score / citation 可解释
向量库只预留接口
知识文档控制在 6 到 10 篇，保证内容质量
测试围绕 SQLi / XSS / Path Traversal 三类典型攻击
```
