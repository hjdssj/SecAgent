# SecRAG Agent 第七阶段任务规划

## 一、阶段目标

第七阶段目标是建设前端 SOC 控制台，让系统分析结果从 Swagger 和命令行升级为可视化安全运营界面。

第六阶段完成后，后端已经具备：

```text
WAF 日志接入
Redis 实时事件流
攻击识别
RAG 知识增强
威胁情报
历史记忆
Markdown 报告
```

第七阶段要新增：

```text
Frontend SOC Console
  -> 告警列表
  -> 告警详情
  -> 风险概览
  -> Markdown 报告展示
  -> 实时刷新
```

## 二、阶段价值

前端完成后，项目更适合演示：

```text
左侧看告警列表
中间看事件详情
右侧看证据、ATT&CK、威胁情报、处置建议
顶部看风险统计
```

这会让项目从后端能力展示变成一个完整产品雏形。

## 三、前后端链路

```text
WAF
  -> Redis
  -> FastAPI
  -> /api/alerts/recent
  -> Frontend
  -> SOC Dashboard
```

可选实时链路：

```text
FastAPI WebSocket
  -> Frontend live updates
```

MVP 先用轮询即可：

```text
每 3 到 5 秒请求 /api/alerts/recent
```

## 四、推荐技术选型

```text
React + TypeScript + Vite
Tailwind CSS
lucide-react
react-markdown
```

如果项目中已有前端框架，优先遵循已有框架。

## 六、页面设计

第七阶段不要做营销首页，直接进入 SOC 工作台。

建议布局：

```text
顶部：系统名称 + 刷新状态 + 风险统计
左侧：告警列表
右侧：告警详情
底部或详情内：Markdown 报告
```

核心信息：

```text
attack_type
risk_level
risk_score
source_ip
target
confidence
evidence
mitre_attack
recommendations
report_markdown
```

## 七、推荐开发顺序

### Step 1：创建 frontend 项目

在项目根目录创建：

```text
frontend/
```

使用 Vite 初始化 React + TypeScript。

### Step 2：定义前端类型

文件：

```text
frontend/src/types/alert.ts
```

类型字段与后端 `SecurityAlert` 保持一致。

### Step 3：封装 API

文件：

```text
frontend/src/api/alerts.ts
```

请求：

```text
GET http://127.0.0.1:8000/api/alerts/recent
```

### Step 4：实现告警列表

组件：

```text
AlertTable.tsx
RiskBadge.tsx
```

列表展示：

```text
风险等级
攻击类型
源 IP
目标路径
置信度
```

### Step 5：实现告警详情

组件：

```text
AlertDetail.tsx
ReportPanel.tsx
```

展示：

```text
关键证据
ATT&CK 映射
处置建议
Markdown 报告
```

### Step 6：实现统计栏

组件：

```text
StatBar.tsx
```

统计：

```text
总告警数
critical 数
high 数
攻击源 IP 数
最新更新时间
```

### Step 7：接入轮询刷新

每 3 到 5 秒自动请求最近告警。

注意：

```text
不要刷新时清空页面
不要让选中的告警因为刷新丢失
```

### Step 8：后端开启 CORS

文件：

```text
backend/app/main.py
```

允许前端开发地址访问：

```text
http://127.0.0.1:5173
http://localhost:5173
```

## 八、验收标准

满足以下条件，即认为第七阶段完成：

```text
前端可以启动
前端可以请求 /api/alerts/recent
页面可以展示告警列表
点击告警可以展示详情
风险等级有明确视觉区分
Markdown 报告可以正常渲染
页面可以自动刷新最新告警
SQLi / XSS / Path Traversal 告警都能在页面中看到
```

## 九、暂不实现

第七阶段先不做：

```text
登录注册
权限系统
多租户
复杂图表大屏
WebSocket 强实时
告警处置工作流
```

MVP 目标是一个清晰、稳定、可演示的 SOC 控制台。

## 十、风险点

### 1. 前后端字段不一致

前端类型必须跟后端模型对齐，避免展示空值。

### 2. CORS 问题

如果前端请求失败，优先检查 FastAPI CORS 配置。

### 3. 页面信息太散

SOC 工具要便于扫描，优先做清晰列表和详情，不要做花哨装饰。
