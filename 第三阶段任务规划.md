# SecRAG Agent 第三阶段任务规划

## 一、阶段目标

第三阶段目标是把第二阶段的“样例日志文件批量回放”升级为“基于 Redis Stream 的实时安全事件流”。

第二阶段已经完成：

```text
data/sample_logs/*.log
  -> ModSecurityParser.parse_line()
  -> SecurityEvent
  -> SecurityAnalysisOrchestrator.analyze()
  -> SecurityAlert
  -> reports/*.md
```

第三阶段要完成：

```text
日志采集器 / 回放脚本
  -> Redis Stream: security:events
  -> 后端 Consumer
  -> SecurityAnalysisOrchestrator
  -> Redis Stream: security:alerts
  -> 告警查询接口
```

这个阶段先不接真实 WAF、LlamaIndex RAG、Milvus 和前端。核心是让系统具备“持续接收事件、持续消费分析”的实时处理雏形。

## 二、为什么第三阶段做 Redis Stream

Redis Stream 是从 Demo 走向工程化的关键过渡。

它能让项目从：

```text
脚本一次性读取日志
```

升级为：

```text
日志事件持续进入队列，后端持续消费分析
```

这样后续接入 WAF 时，只需要把 WAF 日志采集器写入 `security:events`，分析链路不用大改。

## 三、第三阶段核心链路

```text
data/sample_logs/*.log
  ↓
scripts/publish_sample_logs.py
  ↓
Redis Stream: security:events
  ↓
backend/app/services/event_consumer.py
  ↓
SecurityAnalysisOrchestrator
  ↓
Redis Stream: security:alerts
  ↓
GET /api/alerts/recent
```

## 五、模块职责

### 1. redis_client.py

位置：

```text
backend/app/storage/redis_client.py
```

职责：

```text
连接 Redis
写入 security:events
读取 security:events
写入 security:alerts
读取 security:alerts
```

建议封装函数：

```text
get_redis_client()
publish_event(event)
read_events()
publish_alert(alert)
read_recent_alerts()
```

Redis Stream 名称：

```text
security:events
security:alerts
security:deadletter
```

### 2. event_consumer.py

位置：

```text
backend/app/services/event_consumer.py
```

职责：

```text
从 security:events 读取安全事件
转换为 SecurityEvent
调用 SecurityAnalysisOrchestrator.analyze()
将 SecurityAlert 写入 security:alerts
异常事件写入 security:deadletter
```

第一版可以先写同步循环，不急着做复杂 consumer group。

建议核心函数：

```text
consume_once()
consume_forever()
```

### 3. alerts.py

位置：

```text
backend/app/api/alerts.py
```

职责：

```text
提供最近告警查询接口
```

建议接口：

```text
GET /api/alerts/recent
```

返回：

```text
list[SecurityAlert]
```

### 4. publish_sample_logs.py

位置：

```text
scripts/publish_sample_logs.py
```

职责：

```text
读取 data/sample_logs/*.log
使用 ModSecurityParser 解析为 SecurityEvent
写入 Redis Stream: security:events
```

它是第三阶段的“模拟日志采集器”。

后续真实 WAF 接入时，可以把它替换为：

```text
waf_log_collector.py
```

### 5. docker-compose.yml

第三阶段先只需要 Redis：

```text
redis:
  image: redis:7
  ports:
    - "6379:6379"
```

后续再扩展：

```text
Milvus
WAF
Nginx
Frontend
```

## 六、推荐开发顺序

### Step 1：启动 Redis

新增 `docker-compose.yml`。

先只包含 Redis：

```yaml
services:
  redis:
    image: redis:7
    container_name: secrag-redis
    ports:
      - "6379:6379"
```

启动：

```bash
docker compose up -d redis
```

### Step 2：编写 Redis 客户端

新增：

```text
backend/app/storage/redis_client.py
```

先实现最小能力：

```text
连接 Redis
xadd 写入 security:events
xrange 读取 security:alerts
```

第三阶段建议先用 JSON 字符串存储事件和告警。

### Step 3：编写 publish_sample_logs.py

新增：

```text
scripts/publish_sample_logs.py
```

复用第二阶段能力：

```text
load sample logs
ModSecurityParser.parse_line()
SecurityEvent.model_dump_json()
Redis XADD security:events
```

预期输出：

```text
published event: /login from 45.67.89.10
published event: /search from 8.8.8.8
```

### Step 4：编写 event_consumer.py

新增：

```text
backend/app/services/event_consumer.py
```

先实现：

```text
consume_once()
```

逻辑：

```text
读取 security:events
反序列化为 SecurityEvent
调用 orchestrator.analyze()
写入 security:alerts
```

再实现：

```text
consume_forever()
```

用于持续消费。

### Step 5：增加告警查询接口

新增：

```text
backend/app/api/alerts.py
```

接口：

```text
GET /api/alerts/recent
```

返回最近的告警列表。

然后在 `main.py` 注册：

```text
alerts_router
```

### Step 6：端到端测试

推荐测试顺序：

```bash
docker compose up -d redis
python scripts/publish_sample_logs.py
python -m backend.app.services.event_consumer
uvicorn app.main:app --reload --port 8000
```

也可以先不启动 FastAPI，直接从 Redis 中看 `security:alerts` 是否有数据。

## 七、第三阶段验收标准

满足以下条件，即认为第三阶段完成：

1. Redis 能通过 Docker Compose 启动。
2. `publish_sample_logs.py` 能把样例日志写入 `security:events`。
3. `event_consumer.py` 能从 `security:events` 消费事件。
4. Consumer 能复用第一阶段 Orchestrator 生成 `SecurityAlert`。
5. 告警能写入 `security:alerts`。
6. `GET /api/alerts/recent` 能返回最近告警。
7. SQLi、XSS、路径穿越、正常请求都能通过 Redis 流完成分析。

## 九、风险点

### 1. JSON 序列化

`datetime`、Pydantic 模型、嵌套对象写入 Redis 前需要序列化。

建议：

```text
event.model_dump_json()
alert.model_dump_json()
```

读取时：

```text
SecurityEvent.model_validate_json()
SecurityAlert.model_validate_json()
```

### 2. 重复消费

第一版如果用 `XRANGE`，可能会重复读取旧消息。

MVP 可以先接受重复消费，后续再引入：

```text
XREAD
last_id
consumer group
```

### 3. Redis 未启动

脚本和后端需要对 Redis 连接失败给出清晰错误。

### 4. 报告文件重复生成

如果 Consumer 每次生成新的 `alert_id`，多次消费会生成多份报告。

MVP 可以接受，后续再做幂等。

## 十、第三阶段完成后的下一步

第三阶段完成后，进入第四阶段：真实 WAF 日志接入。

第四阶段目标：

```text
Nginx + ModSecurity + OWASP CRS
  -> WAF audit log
  -> waf_log_collector.py
  -> Redis Stream: security:events
  -> Consumer
  -> security:alerts
```

第四阶段完成后，项目就会从“样例日志实时流”升级为“准真实 WAF 联动分析系统”。
